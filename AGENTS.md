# BookWise 项目记忆

## 验收文档约定

每完成一个 Step，必须将执行报告写入 `ACCEPTANCE.md` 验收文档中对应章节。用户需要对着该文档 review 每一步的完成情况。

验收文档格式：
- 每个 Step 独占一节
- 记录实际执行的变更、遇到的问题、与计划的偏差
- 列出该 Step 的验收标准 checklist，完成后打勾
- 附上当次 lint/build 结果截图或摘要

## 任务结束汇报规范

每次任务结束，除了实际代码或文档修改外，还必须给出一份像真实开发者提交阶段结果时的汇报。汇报目标不是“简单说做完了”，而是让人快速判断这次改动是否达成目标、验证到了什么、还剩什么风险。

汇报最低要求：
- 先说明本次任务目标和完成范围
- 说明实际改了哪些模块、页面、接口或基础设施
- 明确哪些内容是本次顺手修复的，哪些内容故意留到后续 Step
- 说明验证结果，至少包括本次是否运行了 `lint`、`build`、手动检查
- 如果验证失败，必须写清楚失败点属于“本次引入的问题”还是“项目既有问题”
- 明确记录阻塞项、遗留问题和下一步建议

推荐汇报结构：
- `目标`
- `实现内容`
- `验证结果`
- `遗留问题 / 风险`
- `阶段结论`

汇报风格要求：
- 使用中文
- 写成开发阶段汇报，而不是聊天口吻
- 结论先行，细节支撑
- 不隐藏失败项，不把历史问题误写成“本次完成”
- 如果只完成部分任务，必须明确写“完成到哪一步”

与 `ACCEPTANCE.md` 的关系：
- 如果本次任务对应某个 Step，汇报内容必须同步写入 `ACCEPTANCE.md`
- `ACCEPTANCE.md` 中的记录应比终端里的最终回复更完整、更适合后续 review
- 最终回复可以简短，但不能和 `ACCEPTANCE.md` 的结论矛盾

## 技术栈

Next.js 16 + React 19 + TypeScript + Prisma + MySQL + NextAuth v5 + Tailwind CSS

## 开发规范

- 新增依赖需经确认
- **Import 规范（必须）**：新代码统一使用 `"@/lib/db/prisma"` 导入 Prisma client，禁止新建独立 PrismaClient 实例
- 用户端页面统一使用 `UserLayout` + `Navbar`
- 状态管理使用 TanStack Query + Zustand
- API 返回统一 JSON，catch 必须返回 500
